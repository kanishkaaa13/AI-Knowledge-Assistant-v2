# Phase 9: Persistent Chat Memory Architecture

## Overview

The chat memory system is designed around durable `conversations` and `messages` records that already exist in PostgreSQL. Phase 9 adds a dedicated service and API layer so the frontend can treat memory as a first-class feature instead of temporary UI state.

## Backend Layers

### Repository Layer

- `ConversationRepository`
  - Lists, searches, and resolves user-owned conversations.
  - Exposes helper lookups for message counts.
- `MessageRepository`
  - Lists ordered messages per conversation.
  - Generates the next `sequence_number`.
  - Resolves the last stored message for previews and summary updates.

### Service Layer

- `ChatMemoryService`
  - Guards ownership checks.
  - Creates blank or seeded conversations.
  - Appends user and assistant messages.
  - Renames and deletes conversations.
  - Auto-generates conversation titles from the first meaningful user prompt.
  - Updates summaries after assistant replies.

### API Layer

- `GET /conversations`
  - Searchable conversation history for the sidebar.
- `POST /conversations`
  - Creates a reusable blank conversation shell or an initialized thread.
- `GET /conversations/{conversation_id}`
  - Loads the full message history for reopening older chats.
- `PATCH /conversations/{conversation_id}`
  - Renames a conversation.
- `DELETE /conversations/{conversation_id}`
  - Removes a conversation and cascades message deletion.
- `POST /assistant/query`
- `POST /assistant/chat/stream`
  - Both routes now persist chat memory automatically while generating answers.

## Persistence Flow

1. The frontend opens a new draft chat without creating clutter in the database.
2. On the first real send, the frontend creates a lightweight conversation shell.
3. The assistant route stores the user message, retrieves context, and generates the answer.
4. The final assistant reply is stored as a message.
5. The conversation title and summary are updated so the sidebar stays useful and searchable.

## Frontend Memory Model

- Sidebar history is fetched from `/conversations`.
- Full thread contents are fetched lazily from `/conversations/{id}` when a chat is opened.
- Search terms are passed to the backend for scalable filtering.
- Conversations are grouped into:
  - Today
  - Yesterday
  - Last 7 days
  - Last 30 days
  - Older
- Rename and delete actions are routed through dedicated mutations.

## Why This Scales

- History lists stay lightweight because the sidebar only loads conversation metadata.
- Full message bodies are loaded on demand.
- Backend ownership rules stay centralized in one memory service.
- The architecture supports future additions such as:
  - pinned conversations
  - archived conversations
  - autogenerated summaries
  - message feedback
  - token usage analytics
